# PROJECT MODIFICATION RULES

Your task is to improve only the frontend appearance and connect the existing functionality to the frontend. The project is already working, so you must preserve all existing scripts, logic, features, files, and functionality.

## Mandatory Rules

1. First, familiarize yourself with the entire project structure, existing files, frontend components, scripts, APIs, routes, dependencies, and current functionality before making any changes.

2. Do not delete, rename, move, replace, or recreate any file or folder.

3. Do not delete, remove, disable, rewrite, or simplify any existing code.

4. Do not remove or modify any existing feature, function, script, API, route, database operation, configuration, dependency, event handler, or business logic.

5. Do not refactor working code unless I explicitly instruct you to do so.

6. Do not change how the project is currently implemented based on your personal opinion, preferred architecture, coding style, framework preference, or design philosophy.

7. Do not introduce alternative implementations when the current implementation is already working.

8. Do not perform bonus work, additional improvements, cleanup, optimization, restructuring, modernization, or unrelated fixes.

9. Only make the exact changes that I request. Do not expand the scope of the task.

10. Your primary responsibility is to improve the frontend design, layout, responsiveness, accessibility, and user experience without affecting existing functionality.

11. Connect the frontend only to functionality that already exists in the project.

12. Preserve all existing function names, variable names, file paths, imports, exports, API endpoints, request formats, response formats, database structures, and integration contracts.

13. Do not create mock functionality, fake data, placeholder logic, or duplicate implementations when working functionality already exists.

14. Do not replace existing backend logic with frontend-only logic.

15. Do not change dependencies, package versions, configuration files, environment variables, build tools, or deployment settings unless I explicitly request it.

16. Do not remove code because it appears unused. It may be required elsewhere in the project.

17. Make the smallest possible changes needed to complete the requested frontend task.

18. Before editing a file, identify exactly why the file must be changed and confirm that the change is directly related to my request.

19. After making changes, verify that all existing functionality still works exactly as it did before.

20. Do not leave incomplete components, broken imports, disconnected buttons, non-functional forms, console errors, build errors, or unfinished code.

21. If a frontend element needs functionality, connect it to the existing script, function, route, API, or event handler. Do not rewrite the underlying functionality.

22. If my instruction is unclear, stop and ask for clarification instead of making assumptions.

23. Never make destructive changes without my explicit written permission.

## Required Workflow

### Step 1: Understand the Project

Inspect and understand:

* The complete file and folder structure
* Existing frontend components and styling
* Existing scripts and application logic
* Existing routes and API endpoints
* Existing event handlers
* Existing integrations
* How data moves between the frontend and backend
* Which features are already working

Do not modify anything during this inspection stage.

### Step 2: Identify the Minimum Required Changes

Determine the smallest set of frontend files that must be edited to complete my exact request.

Do not include unrelated files.

### Step 3: Preserve Existing Functionality

Before making changes, ensure that:

* No working code will be removed
* No file will be deleted
* No feature will be disabled
* No existing implementation will be replaced
* No backend or script logic will be altered unnecessarily

### Step 4: Improve the Frontend Only

You may improve:

* Visual design
* Layout
* Spacing
* Typography
* Colors
* Responsiveness
* Component presentation
* Navigation presentation
* Form presentation
* Accessibility
* User feedback and loading states

You must not alter the underlying functionality unless a minimal connection is required to link an existing function to the frontend.

### Step 5: Connect Existing Functionality

When connecting frontend elements:

* Use the existing functions and scripts
* Use the existing API routes
* Use the existing request and response formats
* Preserve existing validation and error handling
* Do not duplicate logic
* Do not invent new functionality

### Step 6: Verify the Changes

Confirm that:

* The project still builds and runs
* All existing features remain available
* Existing scripts still work
* Buttons and forms remain functional
* API connections still work
* No files or code were removed
* No unrelated changes were introduced
* The changes match only what I requested

## Final Response Requirements

After completing the task, report only:

1. The files you changed
2. The exact frontend changes made
3. How the existing functionality was connected
4. Confirmation that no files, features, scripts, or working code were removed
5. Any issue that prevented you from following my instructions exactly

Do not claim that something works unless you have verified it.

## Core Instruction

Improve the frontend without tampering with the working scripts, logic, architecture, files, or functionality. Preserve everything that already works. Only connect existing functionality to the frontend and only make the changes I explicitly request.
